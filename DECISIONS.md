# FilmBluesia Migration Decisions

## Migration Date

2026-06-13

## Current Architecture Summary

- Repository is already in a partial Next.js App Router shape under `app/`, `components/`, and `lib/`.
- Required legacy inspection files `docs/FILE_MAP.md`, `astro.config.mjs`, and `wrangler.jsonc` are absent in this checkout.
- The pre-migration Next version still carried VPS/Cloudflare-era assumptions: filesystem cache, `/api/image` Sharp image proxy, `/api/cache/status`, `/api/cache/warmup`, request proxy cache headers, and Cloudflare-specific CDN headers.

## Target Architecture Summary

- Next.js App Router on Vercel Free.
- Server Components render shared metadata pages with ISR and Next server `fetch` revalidation.
- Client Components handle HLS playback, iframe facade, search suggestions, localStorage favorites/history, and bottom navigation state.
- OPhim normalization remains in framework-neutral `lib/` modules.
- User-specific state remains in browser `localStorage`.
- Video and poster media are never proxied through Vercel.

## Kept Features

- Home route with spotlight and movie sections.
- Category/list pages with quick filters and pagination.
- Search page plus debounced search suggestions.
- Movie detail page with metadata, images, rating badges, local actions, episode list, and watch CTA.
- Watch page with episode/server selection, OPhim HLS, Vidsrc/Vsembed click-to-load iframe facade, and watch history recording.
- Favorites and history pages using localStorage.
- Mobile-first `max-w-[720px]` app shell and bottom navigation.
- Manifest, icons, share metadata, robots, and sitemap.

## Changed Implementation Details

- On 2026-06-14, production frontend URL defaults were moved from `https://film.bluesia.net` to `https://phim.bluesia.net` for metadata base, robots, sitemap, package identity, and OPhim request User-Agent. Deploy-specific overrides remain controlled by `NEXT_PUBLIC_SITE_URL`.
- Added `.env.example` for Vercel/GitHub auto deploy with `NEXT_PUBLIC_SITE_URL=https://phim.bluesia.net`, `OPHIM_BASE_URL=https://ophim1.com`, and optional Vidsrc/Vsembed variables.
- Added `.env.production` with non-secret production defaults, including `NEXT_PUBLIC_IMAGE_CACHE_URL=https://img.bluesia.net`, because production deploys without that build-time public variable fallback to direct upstream OPhim image URLs and create heavy poster/backdrop loads.
- Defined the external image-cache frontend contract as `https://img.bluesia.net/i/m/<sha256>.webp?url=<encoded-normalized-upstream-image-url>&sig=v1.<hmac-sha256-hex>` and `https://img.bluesia.net/i/d/<sha256>.webp?url=<encoded-normalized-upstream-image-url>&sig=v1.<hmac-sha256-hex>`. When `NEXT_PUBLIC_IMAGE_CACHE_URL` and server-only `IMAGE_CACHE_SIGNING_SECRET` are set, trusted OPhim poster/backdrop URLs are signed server-side before native `<img>`/`<picture>` rendering; Vercel still does not fetch, validate, transform, cache, or proxy image binaries. Legacy `/image?url=` is VPS backward compatibility only, still requires `sig`, and is not generated by frontend helpers.
- On 2026-06-15, server-only signing moved into `lib/image-signing.server.ts`, which imports `server-only`, reads `IMAGE_CACHE_SIGNING_SECRET`, and uses Node HMAC-SHA256. Signing is opt-in from trusted OPhim server-side movie data/API response mapping, with no public arbitrary image signing endpoint and no strict manual CDN host allowlist that can break new OPhim image hosts. Client Components no longer generate cache URLs; they only render `posterSources` and `thumbSources` prepared by server pages or trusted `/api/ophim/*` metadata handlers.
- `lib/images.ts` remains client-safe and contains no secret access. It trims image sources, accepts only HTTP(S) upstream URLs, lowercases only the hostname, preserves protocol/path/query casing, removes empty fragments, and exposes deterministic SHA-256 for client-safe checks. Server signing hashes the same normalized upstream URL with Node `crypto`. The variant is represented only by `/i/m/` or `/i/d/`; the hash never includes variant, width, quality, DPR, frontend host, or runtime options.
- The only frontend-generated image variants are `m` and `d`: `m` maps to VPS max width 480px WebP quality 75, and `d` maps to VPS max width 960px WebP quality 75. The frontend does not send `w`, `q`, `dpr`, `format`, AVIF, DPR variants, arbitrary width lists, or per-component transformation options.
- Reset localStorage keys to `phim.bluesia.net:*`. Old `film.bluesia.net:*` and `bluesia:*` browser data is intentionally not migrated.
- Updated project packages to current stable compatible versions on 2026-06-13: Next.js 16.2.9, React/React DOM 19.2.7, Tailwind CSS/PostCSS plugin 4.3.1, PostCSS 8.5.15, lucide-react 1.18.0, and current type packages.
- Kept ESLint on 9.39.4 instead of npm `latest` 10.5.0 because `eslint-config-next` 16.2.9's bundled `eslint-plugin-react` fails under ESLint 10 with `contextOrFilename.getFilename is not a function`.
- Kept `critters` at 0.0.25 because the registry `latest` tag currently points backward to 0.0.23; downgrading would not be a stable update.
- Poster and backdrop images now use native `<img>`/`<picture>` URLs instead of `/api/image` proxy URLs or Vercel `next/image` optimization. When `NEXT_PUBLIC_IMAGE_CACHE_URL` is set, the frontend rewrites safe HTTP(S) image URLs to the external image cache. Native `<img>` lint warnings are intentional for movie art.
- Vercel `next/image` remote optimization was disabled on 2026-06-14 after the free team reached 75% of the included Image Optimization Cache Writes quota. The app now renders upstream poster/backdrop URLs with native `<img>`, stable dimensions, eager fetch priority only for true LCP images, and lazy loading for poster grids/search/player facades. `next.config.mjs` intentionally has no `images` optimizer configuration so remote movie art does not create `/_next/image` cache writes.
- Removed active `/api/image` route and the direct `sharp` dependency to avoid high-volume Function invocations, Image Optimization-like quota pressure, bandwidth amplification, and ephemeral filesystem cache writes.
- Removed active `/api/cache/status` and `/api/cache/warmup` routes because cache warmup/inspection was a VPS/Docker operation and would spend Vercel Functions without durable storage.
- Removed `lib/cache.ts` filesystem cache. Metadata now relies on Next server `fetch` revalidation.
- Removed `proxy.ts`; page cache headers are handled by route/page revalidation and `next.config.mjs`, avoiding request-wide proxy execution.
- Removed Cloudflare-specific CDN response headers from active runtime code.
- Added `returnTo` context threading from section/list/search/hero links into movie/watch routes.
- Watch episode links now use replacement navigation to reduce browser history pollution during episode switches.
- Watch episode resolution now matches explicit episode keys, slugs, names, and filenames before using legacy zero-based numeric index fallback so upstream `ep=1`/`slug=1` opens episode 1 instead of episode 2.
- Bottom nav active state now considers `returnTo` while on movie/watch routes.
- Bottom nav is wrapped in a root-layout Suspense boundary because it reads search params and otherwise blocks static 404 prerendering in Next.js.
- Added lightweight `app/robots.ts` and `app/sitemap.ts` for core routes only; dynamic movie/search pages are not mass-generated.

## Removed Cloudflare-Only Assumptions

- No Cloudflare KV/R2/Cache API is required.
- No Cloudflare-specific response headers remain in active app code.
- No Docker cache warmer is part of the Vercel runtime.
- No active server-side image optimizer/cache route remains.

## Vercel Free Limit Mitigations

- Avoids high-volume poster Function calls by using direct upstream image URLs or the two fixed external `img.bluesia.net` variants, never Vercel image proxy routes.
- Avoids Vercel Image Optimization Cache Writes by not using `next/image` for remote movie posters/backdrops.
- Avoids video/HLS/embed proxying completely.
- Avoids write-heavy server storage and databases.
- Avoids cron/background jobs/warmers.
- Avoids request-wide proxy execution.
- Avoids build-time generation of arbitrary movie pages.
- Search remains runtime/client initiated and cached by the small JSON route handler.
- Metadata fetches are time-bounded and revalidated conservatively.

Fast Origin Transfer remains a close secondary risk, so future changes should also avoid increasing origin egress, video proxying, uncached metadata fetches, or request-wide edge/function work without approval.

## Open Risks

- If `NEXT_PUBLIC_IMAGE_CACHE_URL` is unset, poster/backdrop rendering falls back to direct upstream/CDN image URLs. If `IMAGE_CACHE_SIGNING_SECRET` is missing, the server logs one warning and falls back to the normalized upstream URL instead of silently generating unsigned `img.bluesia.net` URLs.
- Without filesystem stale fallback, a cold request can fail if OPhim is unavailable and there is no valid Next fetch cache entry.
- `compose.yaml` and `Dockerfile` remain for non-Vercel workflows and may still describe Docker behavior not used by Vercel.
